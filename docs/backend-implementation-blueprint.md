# MerchantHub — Backend Implementation Blueprint

**Version:** 1.0  
**Status:** Draft  
**Phase:** 2 (Implementation Blueprint)  
**Last Updated:** July 1, 2026

---

## 1. Purpose

This document bridges the gap between architecture and implementation. It describes **how** the MerchantHub backend will be constructed in Django, without containing actual code.

The existing architecture documents form the source of truth:
- **Project Constitution** (PRD) — product requirements, scope, vision
- **Domain Model** — entities, aggregate roots, invariants, lifecycle
- **Database Schema** — tables, columns, constraints, indexes
- **Database Decisions** — trade-offs, precision, timezone, soft-delete strategy
- **Relationships** — entity dependencies, ownership boundaries, cascade rules

This blueprint translates those documents into an implementable plan. Developers reading this document should understand exactly which files to create, which classes to define, and how the pieces connect — without needing to rediscover the architecture.

## 2. Project Structure

The project follows Django's standard project layout. The root structure already exists:

```
MerchantHub/                          # Project root (git repository root)
├── apps/                             # All Django applications
│   ├── accounts/                     # User management and authentication
│   ├── common/                       # Shared base classes, utilities, cross-cutting
│   ├── workspaces/                   # Multi-tenancy, workspace management
│   ├── inventory/                    # Products, categories, stock tracking
│   ├── customers/                    # Customer profiles and history
│   ├── sales/                        # Sales transactions and line items
│   ├── payments/                     # Payment recording and reconciliation
│   ├── expenses/                     # Expense tracking and categorization
│   └── dashboard/                    # Aggregated views and KPIs (no models)
├── config/                           # Django project configuration
│   ├── settings.py                   # All settings (exists, will be extended)
│   ├── urls.py                       # Root URL configuration (exists, will be extended)
│   ├── wsgi.py                       # WSGI entry point
│   └── asgi.py                       # ASGI entry point (future websocket support)
├── docs/                             # Architecture and design documentation
├── manage.py                         # Django management script
└── venv/                             # Python virtual environment
```

### Directory Responsibilities

| Path | Responsibility |
|---|---|
| `apps/common/` | Abstract base models (`BaseModel`, `WorkspaceScopedModel`, `SoftDeleteModel`), custom model fields, shared querysets, utility functions, cross-cutting middleware, base DRF classes. |
| `apps/accounts/` | Custom User model, JWT authentication views (register, login, refresh, password reset), profile management. No workspace logic. |
| `apps/workspaces/` | Workspace CRUD, membership management, role definitions, business profile. This is the tenant foundation. |
| `apps/inventory/` | Category CRUD, Product CRUD, stock management, inventory movement creation, low-stock queries. |
| `apps/customers/` | Customer CRUD, search, computed metrics (total visits, total spend). |
| `apps/sales/` | Sale creation, sale items, automatic inventory deduction, sale history, status management. |
| `apps/payments/` | Payment recording against sales, payment method tracking, outstanding balance queries. |
| `apps/expenses/` | Expense CRUD, categorization, date-range queries. |
| `apps/dashboard/` | Read-only KPI aggregation across all modules. No models of its own. |
| `config/` | Settings, root URL routing, WSGI/ASGI config. No business logic. |

### 3.1 accounts

| Attribute | Value |
|---|---|
| **Responsibility** | User identity, authentication, profile management. |
| **Owned models** | `User` (extends `AbstractUser`, UUID PK override) |
| **Services** | `RegistrationService` — creates user account via `accounts` and delegates default workspace + owner membership creation to `workspaces.WorkspaceService.create_default_workspace()` in one atomic transaction |
| **Serializers** | `RegisterSerializer`, `LoginSerializer`, `UserProfileSerializer`, `PasswordResetSerializer` |
| **Views** | `RegisterView`, `LoginView`, `TokenRefreshView`, `ProfileViewSet`, `PasswordResetView` |
| **Permissions** | `IsAuthenticated` (DRF default) for profile; `AllowAny` for registration/login |
| **Signals** | None in MVP. Future: `post_save` for audit log creation. |
| **Dependencies** | `workspaces` (service-level delegation — calls `WorkspaceService.create_default_workspace()` during registration). Depends on Django's `auth` and `contenttypes` frameworks. |

**Implementation notes:**
- The `User` model already exists with `email`, `phone_number`, and `is_merchant`. It needs a UUID PK override and `db_table` set to `'accounts_user'`.
- Registration must be wrapped in `transaction.atomic()` because it creates a User and delegates workspace creation atomically.
- `accounts` never imports workspace models directly. It calls `WorkspaceService.create_default_workspace()` via a service-layer interface, not by importing models. This keeps module ownership boundaries clean.
- JWT authentication uses `djangorestframework-simplejwt` (already listed in settings).

| **Owned models** | `Workspace`, `WorkspaceMembership`, `BusinessProfile`, `Role`, `Permission` |

- `BusinessProfile` is created automatically when a `Workspace` is created (via signal or within the registration service). It has no independent create/delete endpoints — it is always managed through the Workspace.

### 3.3 inventory

| Attribute | Value |
|---|---|
| **Responsibility** | Product catalog management, stock tracking, inventory movements. |
| **Owned models** | `Category`, `Product`, `InventoryMovement` |
| **Services** | `ProductService` — CRUD with SKU uniqueness enforcement; `InventoryService` — stock adjustments, low-stock queries; `MovementService` — create initial/adjustment/return movements, ensure immutability |
| **Serializers** | `CategorySerializer`, `ProductSerializer`, `ProductListSerializer` (lightweight for listing), `InventoryMovementSerializer` (read-only for movements) |
| **Views** | `CategoryViewSet`, `ProductViewSet`, `InventoryMovementViewSet` (read-only, with product filter), `LowStockProductView` (custom endpoint) |
| **Permissions** | `IsWorkspaceMember` for read; `role >= manager` for write |
| **Signals** | None. Inventory movements are created explicitly by services, not automatically by signals. |
| **Dependencies** | `workspaces.Workspace` (scoping), `sales.Sale` (optional FK on InventoryMovement) |

**Implementation notes:**
- `Product.current_stock` is a denormalized field (see database-decisions.md). It is updated atomically within the same transaction that creates InventoryMovements.
- `InventoryMovement` is append-only. No update or delete operations are exposed.
- `sale_id` on InventoryMovement is nullable and only populated for `sale` or `return` movement types.
- Low-stock query: `Product.objects.for_workspace(ws).filter(current_stock__lte=F('low_stock_threshold'), low_stock_threshold__isnull=False)`
- **Circular import mitigation:** `InventoryMovement.sale_id` references `sales.Sale`. Because `sales` also references `inventory.Product` via `SaleItem`, use Django's **lazy string-based FK** (`ForeignKey('sales.Sale', ...)`). Do not `from sales.models import Sale` at the top of `inventory/models.py`.

### 3.5 sales

| Attribute | Value |
|---|---|
| **Responsibility** | Sale transactions, line items, automatic inventory deduction, status management. |
| **Owned models** | `Sale`, `SaleItem` |
| **Services** | `SaleService` — the most critical service in the system. Handles sale creation inside `transaction.atomic()` with `select_for_update()` on Product rows. Computes `total_amount` from line items, creates InventoryMovements, updates Customer metrics. Payments are recorded separately via `PaymentService.record_payment()`. |
| **Serializers** | `SaleSerializer`, `SaleCreateSerializer` (accepts nested line items), `SaleListSerializer` (lightweight), `SaleItemSerializer` |
| **Views** | `SaleViewSet` with date range filtering, product/customer search |
| **Permissions** | `IsWorkspaceMember` for read; all roles (including `staff`) can create sales |
| **Signals** | None. All cross-aggregate operations are handled explicitly in SaleService. |
| **Dependencies** | `workspaces.Workspace` (scoping), `inventory.Product` (SaleItem FK, stock deduction), `customers.Customer` (optional FK, metric update), `accounts.User` (recorded_by) |

**Implementation notes:**
- Sale creation is the most complex operation. The service method:
  1. Locks all Product rows with `select_for_update()`.
  2. Verifies sufficient stock for every line item.
  3. Creates the Sale record.
  4. Creates SaleItem records.
  5. Deducts stock on each Product.
  6. Creates InventoryMovement records with `running_balance`.
  7. Updates Customer computed metrics (if customer_id provided).
  8. Sets Sale.status to `pending`.
- If any step fails, the entire transaction rolls back.
- `Sale.paid_amount` is denormalized. It starts at 0 and is incremented by the payments module.
- **Circular import mitigation:** `SaleItem.product_id` references `inventory.Product`. Because `inventory` also references `sales.Sale` via `InventoryMovement`, use Django's **lazy string-based FK** (`ForeignKey('inventory.Product', ...)`) to avoid circular import errors at Python module load time.
- **Payments are recorded separately.** Sale creation does not record payments. The sale is created in `pending` status. A client records one or more payments via `PaymentService.record_payment()`, which transitions the sale to `partially_paid` or `completed`.

### 3.6 payments

| Attribute | Value |
|---|---|
| **Responsibility** | Payment recording against sales, payment method tracking, outstanding balance queries. |
| **Owned models** | `Payment` |
| **Services** | `PaymentService` — records a payment, validates it doesn't exceed sale total, updates Sale.paid_amount and Sale.status atomically |
| **Serializers** | `PaymentSerializer`, `PaymentCreateSerializer` |
| **Views** | `PaymentViewSet` with sale filter, date range; `OutstandingBalanceView` (sales with remaining balance) |
| **Permissions** | `IsWorkspaceMember` for read; all roles can record payments |
| **Signals** | None. Payment → Sale updates are explicit in PaymentService. |
| **Dependencies** | `workspaces.Workspace` (scoping), `sales.Sale` (FK, PROTECT), `customers.Customer` (optional FK) |

**Implementation notes:**
- `Payment` is a dependent entity within the Sale aggregate in MVP (see domain model). It cannot exist without a parent Sale.
- Payment recording must lock the Sale row (via `select_for_update()`) to prevent concurrent payments from exceeding `total_amount`.
- The service updates `Sale.paid_amount` and then derives `Sale.status`:
  - `paid_amount >= total_amount` → `completed`
  - `paid_amount > 0` → `partially_paid`
  - `paid_amount == 0` → `pending`
- Multiple payments can be applied to a single sale (split payments across cash, card, and bank transfer).

## 4. Shared Architecture

### 4.1 Base Model Hierarchy

The abstract base models live in `apps/common/models/base.py` (or within `apps/common/models.py`).

**Inheritance chain:**

```
django.db.models.Model
  └── UUIDModel (Abstract)
        └── TimeStampedModel (Abstract)
              └── BaseModel (Abstract) = UUIDModel + TimeStampedModel
                    └── SoftDeleteModel (Abstract) = BaseModel + deleted_at
                          └── WorkspaceScopedModel (Abstract) = SoftDeleteModel + workspace FK
```

| Class | Fields | Purpose |
|---|---|---|
| `UUIDModel` | `id = UUIDField(primary_key=True, default=uuid.uuid4, editable=False)` | UUID PK for all entities. |
| `TimeStampedModel` | `created_at`, `updated_at` | Automatic timestamp tracking. |
| `BaseModel` | (inherits both above) | Universal base for all MerchantHub models. |
| `SoftDeleteModel` | `deleted_at` | Adds soft-delete capability. |
| `WorkspaceScopedModel` | `workspace = ForeignKey(Workspace)` | Adds multi-tenant scoping. |

**Which models use which base:**

| Concrete Model | Base Classes |
|---|---|
| `User` | `AbstractUser` (Django) |
| `Workspace` | `BaseModel`, `SoftDeleteModel` |
| `BusinessProfile` | `BaseModel` |
| `WorkspaceMembership` | `BaseModel` |
| `Role` | `BaseModel` |
| `Permission` | `BaseModel` |
| `Category` | `WorkspaceScopedModel`, `SoftDeleteModel` |
| `Product` | `WorkspaceScopedModel`, `SoftDeleteModel` |
| `InventoryMovement` | `BaseModel` |
| `Customer` | `WorkspaceScopedModel`, `SoftDeleteModel` |
| `Sale` | `WorkspaceScopedModel`, `SoftDeleteModel` |
| `SaleItem` | `BaseModel` |
| `Payment` | `WorkspaceScopedModel`, `SoftDeleteModel` |
| `Expense` | `WorkspaceScopedModel`, `SoftDeleteModel` |
| `Notification` | `BaseModel` |
| `AuditLog` | `BaseModel` |

### 4.2 Custom Managers

A `WorkspaceScopedQuerySet` provides automatic workspace filtering:

```python
class WorkspaceScopedQuerySet(models.QuerySet):
    def for_workspace(self, workspace):
        return self.filter(workspace=workspace)
```

Each workspace-scoped model uses this as its default manager:

```python
class Product(models.Model):
    objects = WorkspaceScopedQuerySet.as_manager()
```

This enables: `Product.objects.for_workspace(request.workspace).filter(...)`.

### 4.3 Validators

Shared validators live in `apps/common/validators.py`:

| Validator | Purpose |
|---|---|
| `validate_positive_decimal` | Ensures a decimal value is > 0. |
| `validate_non_negative_decimal` | Ensures a decimal value is >= 0. |
| `validate_iso_currency` | Validates a 3-letter ISO 4217 currency code (basic check). |
| `validate_phone` | Validates phone number format (E.164 optional). |

### 4.4 Permission Classes

Custom DRF permission classes live in `apps/common/permissions.py`:

| Class | Logic |
|---|---|
| `IsAuthenticated` | DRF built-in. Applied globally (except register/login). |

| `IsWorkspaceOwner` | Verifies the user's membership has `role='owner'`. |
| `IsManagerOrAbove` | Verifies `role.level >= 50` (manager or owner). |

### 4.5 Exceptions

Custom exception classes live in `apps/common/exceptions.py`:

| Exception | HTTP Code | Use Case |
|---|---|---|

## 5. Business Logic Placement

### 5.1 Where Logic Lives

| Layer | Responsible for | NEVER responsible for |
|---|---|---|
| **Models** | Field definitions, constraints (`Meta`), string representation, simple computed properties (e.g., `profit = unit_price - cost_price`). | Business logic spanning multiple models, external service calls, cross-aggregate operations, I/O, sending notifications. |
| **Managers/QuerySets** | Reusable query filters (`for_workspace()`, `active()`, `low_stock()`). | Mutations, validation requiring external state. |
| **Services** | Business logic, transaction management, cross-aggregate operations, validation that spans multiple models, denormalized field updates. | HTTP concerns, serialization, permission checks. |
| **Serializers** | Input validation, deserialization, field-level formatting, nested object creation (for simple parent-child only). | Complex cross-model validation, transaction management. |
| **Views/ViewSets** | HTTP orchestration: authentication, permission checks, calling services, returning responses. | Business logic, data manipulation, direct model mutations (always delegate to services). |
| **Signals** | Side effects that should not block the primary operation: audit logging, notification creation. | Critical business logic, transactional mutations. |
| **Middleware** | Cross-cutting concerns: workspace resolution from header, request timing, logging. | Business logic, data access. |
| **Utilities** | Pure functions, helpers, format conversions. | State management, database access. |

### 5.2 The Service Layer Pattern

Services are plain Python classes (not Django models, not DRF views). They accept `user` and `workspace` parameters where needed:

```python
class SaleService:
    @staticmethod
    def create_sale(*, workspace, recorded_by, customer, items):
        # 1. Validate
        # 2. Lock products
        # 3. Check stock
        # 4. Create sale + items
        # 5. Deduct stock
        # 6. Create movements
        # 7. Update customer metrics
        # 8. Set status to pending
        # All inside transaction.atomic()
        pass
```

### 5.3 What Never Goes Where

- **Models never call services.** A model's method should never import and invoke a service from another module.
- **Models never access `request`.** Models are layer-agnostic and know nothing about HTTP.
- **Views never call `Model.objects.create()` directly** for complex operations. They delegate to services.
- **Signals never contain transactional business logic.** If it fails, the operation should roll back — put it in a service instead.
- **Services never return HTTP responses.** They return domain objects or raise domain exceptions.

## 6. Transaction Boundaries

### 6.1 Scope

Use `transaction.atomic()` for any operation that modifies multiple database rows. In Django, this can be applied as a decorator on a service method or as a context manager.

### 6.2 Critical Transactions

**Sale Creation:**
A single `transaction.atomic()` block wraps the entire sale creation flow:
- CREATE Sale → INSERT SaleItems → SELECT ... FOR UPDATE on Products → UPDATE stock → INSERT InventoryMovements → UPDATE Customer metrics.
- If any step fails (insufficient stock, database constraint violation), the entire operation rolls back.
- `select_for_update()` is called **before** the stock check on all affected Product rows.

**Payment Recording:**
- Lock the Sale row with `select_for_update()`.
- Read current `paid_amount` and validate against `total_amount`.
- INSERT the Payment.
- UPDATE `Sale.paid_amount` and `Sale.status`.

**Inventory Adjustment:**
- Lock the Product row.
- INSERT InventoryMovement.
- UPDATE `current_stock`.

### 6.3 Django Utilities

| Utility | When to Use |
|---|---|
| `transaction.atomic()` | Every multi-model operation. |
| `select_for_update()` | Sale creation (on Product rows), payment recording (on Sale row), inventory adjustment (on Product row). |
| `F()` expressions | Atomic field updates: `Product.objects.filter(id=pk).update(current_stock=F('current_stock') - qty)`. |
| `transaction.on_commit()` | For side effects that must only happen after the transaction commits (e.g., sending a notification). |

### 6.4 What Is Not In a Transaction

- Read-only queries (list views, detail views).
- Audit log writes (fire-and-forget; failure must not affect the primary operation).
- Notification creation (can be deferred with `transaction.on_commit()`).

## 8. Permission Strategy

### 8.1 Authentication First

All endpoints (except `register` and `login`) require a valid JWT access token. Authentication is verified by `djangorestframework-simplejwt`'s `JWTAuthentication` class, configured globally in `settings.py`.

### 8.2 Workspace Membership Check

All workspace-scoped endpoints require the `X-Workspace-ID` header. A middleware extracts this header and resolves it to a `Workspace` instance, attaching it to `request.workspace`. Custom permission classes then verify that the authenticated user has an active `WorkspaceMembership` for that workspace.

**Permission evaluation flow:**

```
Request → JWTAuthentication → User identified
         → Middleware reads X-Workspace-ID header
         → Verifies WorkspaceMembership exists and is active
         → Attaches Workspace to request.workspace
         → DRF permission class evaluates role-based access
         → View executes (or 403 Forbidden)
```

### 8.3 Role Hierarchy

| Role | Level | Access |
|---|---|---|
| `owner` | 100 | Full access. Can manage members, delete workspace, view all data, configure settings. |
| `manager` | 50 | Read/write on all business data (products, sales, customers, expenses, payments). Can view member list. Cannot invite/remove members, change roles, or delete workspace. |
| `staff` | 10 | Can record sales and payments, view products and customers, log expenses. Cannot view profit metrics, cost prices, or member list. |

### 8.4 Permission Class Mapping

| Permission Class | Applied To |
|---|---|
| `IsWorkspaceMember` | All workspace-scoped read endpoints. |
| `IsManagerOrAbove` | Product create/update/delete, Customer delete, Expense delete, Category CRUD. |
| `IsWorkspaceOwner` | Workspace update/delete, Member management, Role changes, BusinessProfile update. |
| No custom permission | Sale create (all roles including staff), Payment create, Expense create. |

### 8.5 Staff Data Restrictions

Staff-level users see a restricted view:
- Product list excludes `cost_price`.
- Dashboard excludes `net_profit`.
- Customer metrics show visit count but not total_spend (configurable).

This is enforced at the serializer level: serializer fields are conditionally excluded based on `request.user`'s role in the current workspace.

## 9. Error Handling Strategy

### 9.1 Consistent Response Format

All API errors return a uniform JSON structure:

```json
{
  "error": "Human-readable description of the problem.",
  "code": "MACHINE_READABLE_ERROR_CODE",
  "details": {}
}
```

### 9.2 Error Types

| Scenario | HTTP Status | Error Code | Details |
|---|---|---|---|
| **Validation error** (invalid input) | 400 | `VALIDATION_ERROR` | Field-level errors in `details`. |
| **Business rule violation** (e.g., insufficient stock) | 409 | `INSUFFICIENT_STOCK` | Product IDs, available vs. requested quantities. |
| **Payment exceeds sale total** | 409 | `PAYMENT_EXCEEDS_TOTAL` | Sale total, current paid, attempted payment. |
| **Duplicate SKU** | 409 | `DUPLICATE_SKU` | The conflicting SKU. |
| **Authentication required** | 401 | `NOT_AUTHENTICATED` | (empty) |
| **Permission denied** | 403 | `PERMISSION_DENIED` | Required role level. |
| **Workspace required** | 400 | `WORKSPACE_REQUIRED` | (empty) |
| **Not a member** | 403 | `NOT_WORKSPACE_MEMBER` | Workspace ID. |
| **Not found** | 404 | `NOT_FOUND` | Entity type and ID. |
| **Conflict** (general) | 409 | `CONFLICT` | Description. |

### 9.3 Implementation

- **DRF's built-in validation errors** (`serializers.ValidationError`) return 400 with field-level errors in the standard DRF format.
- **Custom exceptions** (defined in `apps/common/exceptions.py`) are raised by services and caught by a custom DRF exception handler.
- **The custom exception handler** is registered in `settings.py` as `EXCEPTION_HANDLER`. It converts custom exceptions to the standard error format.
- **Unhandled exceptions** (500) return a generic error without exposing internals.

### 9.4 DRF Exception Handler Configuration

```python
# In config/settings.py
REST_FRAMEWORK = {
    'EXCEPTION_HANDLER': 'apps.common.exceptions.custom_exception_handler',
    ...
}
```

The custom handler catches:
- DRF's `APIException` subclasses (ValidationError, NotFound, PermissionDenied, AuthenticationFailed)
- Custom domain exceptions (InsufficientStockError, PaymentExceedsTotalError, etc.)
- Unhandled exceptions → generic 500 response

## 10. Testing Strategy

### 10.1 Test Types

| Type | Scope | What It Tests | Per-App Target |
|---|---|---|---|
| **Unit tests** (`test_models.py`) | Single class/method | Model constraints, validators, custom managers, computed properties. | 1–3 tests per model. |
| **Service tests** (`test_services.py`) | Single service method | Business logic, transaction boundaries, cross-aggregate operations, invariants. | 3–5 tests per service method including success and failure paths. |
| **API tests** (`test_api.py`) | HTTP request → JSON response | Endpoints, permissions, serialization, error responses. | 2–4 tests per endpoint covering success, auth failure, permission denial, and validation errors. |
| **Transaction tests** | Service method with concurrent calls | Rollback behavior, `select_for_update()` correctness, deadlock avoidance. | 1–2 critical tests per transaction boundary. |
| **Permission tests** | API endpoint with different roles | Correct role enforcement for each endpoint. | 1 test per (endpoint, role) combination. |

### 10.2 Fixtures

- `Role` and `Permission` are seeded via data migration, not test fixtures.
- Test factories (using `factory_boy`) create test data with minimal boilerplate.
- Each test creates its own workspace + user + membership to ensure isolation.

### 10.3 What to Test (Required)

| Area | Must Test |
|---|---|
| **Auth** | Registration creates User + Workspace + Membership + BusinessProfile atomically. Duplicate email rejected. Login with valid/invalid credentials. Token refresh. |
| **Workspace** | CRUD with correct ownership checks. Invitation creates membership with correct role. Last owner cannot be removed. |
| **Inventory** | Product CRUD with SKU uniqueness. Stock adjustment. Low-stock query. |
| **Sales** | Sale creation deducts stock. Insufficient stock returns error and rolls back. Sale total equals sum of items. Customer metrics update. |
| **Payments** | Payment recording updates sale status. Payment exceeding total is rejected. Split payments work correctly. |
| **Expenses** | CRUD with category validation. |
| **Permissions** | Staff cannot see cost_price. Manager cannot manage members. Owner can do everything. Non-member gets 403. |

### 10.4 What Not to Test

- Database integrity (trust PostgreSQL CHECK constraints).
- Django/DRF internals (trust the framework).
- Third-party library behavior (trust PyJWT, etc.).

### 10.5 Test Configuration

- Tests use SQLite in-memory database (fast, isolated).
- `AUTH_USER_MODEL` is already configured.
- DRF's `APIClient` or `APITestCase` for API tests.
- `@pytest.mark.django_db` if using pytest.
- Test coverage target: > 80% for business logic modules (sales, inventory, payments).

## 11. Implementation Order

The implementation order is designed to minimize dependency problems. Each step produces a working, testable slice of the system.

```
Foundation
    ↓
Accounts / Auth
    ↓
Workspaces
    ↓
Inventory
    ↓
Customers
    ↓
Sales
    ↓
Payments
    ↓
Expenses
    ↓
Dashboard
    ↓
Integration Testing
    ↓
Deployment
```

### Step-by-Step Rationale

| Step | What to Build | Why This Order |
|---|---|---|
| **1. Foundation** | `apps/common/` base models (BaseModel, SoftDeleteModel, WorkspaceScopedModel), shared querysets, validators, custom exception classes. Update `settings.py` with DRF configuration, drf-spectacular, CORS. | Every other app depends on these base classes and settings. Without this step, nothing compiles. |
| **2. Accounts** | `User` model (UUID override, profile fields), registration serializer, login view, JWT configuration, profile endpoint. | Authentication is required before any workspace-scoped endpoint can be tested. Service-level dependency on `WorkspaceService.create_default_workspace()` interface (defined during foundation). |
| **3. Workspaces** | `Workspace`, `WorkspaceMembership`, `BusinessProfile`, `Role`, `Permission` models. Workspace CRUD views, membership management, business profile. Middleware for `X-Workspace-ID` header. `IsWorkspaceMember`, `IsWorkspaceOwner` permission classes. | All downstream business modules depend on workspace scoping. Registration (Step 2) creates a default workspace, so workspace must be ready. |
| **4. Inventory** | `Category`, `Product`, `InventoryMovement` models. Product CRUD, category CRUD, stock adjustment, inventory movement history. | Products are referenced by Sales and SaleItems. Building inventory first means sales creation can be tested with real products. |
| **5. Customers** | `Customer` model, CRUD views, search. | Customers are referenced by Sales. No dependencies on other modules beyond Workspace. |
| **6. Sales** | `Sale`, `SaleItem` models. The critical `SaleService.create_sale()` with full transaction boundary. Sale history and filtering. | Sales depend on Inventory (Products) and Customers. This is the most complex module — building it after its dependencies exist allows end-to-end testing from the start. |
| **7. Payments** | `Payment` model, payment recording, sale status management, outstanding balance queries. | Payments depend on Sales. Building payments immediately after sales allows the sales → payments flow to be tested end-to-end. |
| **8. Expenses** | `Expense` model, CRUD views, category filtering. | Expenses have no dependencies beyond Workspace. Building them last is a deliberate choice because they are the simplest module. |
| **9. Dashboard** | `DashboardService` with KPI aggregation, activity feed, period comparison. | Dashboard reads from all other modules. Building it last means all data is available for integration testing. |
| **10. Integration Testing** | End-to-end tests covering full user journeys: register → create workspace → add products → create customer → record sale → record payment → check dashboard. | All modules are built. This step validates that the system works as a whole. |
| **11. Deployment** | `settings.py` production configuration, environment variables, Dockerfile, deployment to free-tier hosting. | Deployment is the final step. All code exists and tests pass. |

### Parallelization Opportunities

If multiple developers are working simultaneously:

| Developer A | Developer B |
|---|---|
| Step 1: Foundation | (wait) |
| Step 2: Accounts | Step 3: Workspaces |
| Step 4: Inventory | Step 5: Customers |
| Step 6: Sales | Step 7: Payments |
| Step 8: Expenses | Step 9: Dashboard |
| Steps 10–11: Integration + Deployment | Steps 10–11: Integration + Deployment |

Accounts and Workspaces can be built in parallel if the `WorkspaceService.create_default_workspace()` interface is agreed upon during the Foundation step. Inventory and Customers are independent of each other. Sales and Payments are independent of Expenses and Dashboard.
